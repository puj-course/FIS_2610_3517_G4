\section{Evaluación del producto}
Se produjo gestión documental, mejoras de seguridad y monitoreo. La calidad mejoró por reportes QA y validación de persistencia, aunque quedó incompleta la trazabilidad por PR asignado.

\section{Evaluación del proceso}
Scrum Poker, QA y seguimiento por issues fortalecieron el proceso. El principal problema fue tratar variables de entorno y seguridad tarde.

\section{Evaluación del esfuerzo}
El esfuerzo fue medible con 82 commits y 38 issues cerradas. La participación se distribuyó entre seguridad, documentación, QA y producto.

\section{Postmortem del milestone}
\subsection{Qué salió bien}
Se formalizó QA, se mejoró seguridad y se cerraron las issues del milestone.

\subsection{Qué salió mal}
La gestión de secretos se postergó y algunas HU visuales no tenían criterios responsivos explícitos.

\subsection{Causas raíz}
La causa raíz principal fue configurar variables de entorno tarde, lo que elevó el riesgo de credenciales hardcodeadas.

\subsection{Impacto}
El impacto fue retraso, riesgo de seguridad y presión sobre el pipeline.

\subsection{Acciones correctivas}
Configurar variables desde el día 1, incluir criterios mobile y registrar milestones en PRs.

\subsection{Acciones preventivas y responsables sugeridos}
Configuration Manager debe custodiar entorno; Scrum Master debe revisar riesgos; Product Owner debe exigir criterios responsive; QA Lead debe usar templates reproducibles.

\section{Retrospectiva estrella de mar}
\subsection{Comenzar a hacer}
\begin{itemize}
\item Configurar variables de entorno desde el primer día.
\item Incluir criterios responsivos en HU visuales.
\item Registrar milestone en PRs relevantes.
\end{itemize}

\subsection{Más de}
\begin{itemize}
\item Aplicar Scrum Poker en historias inciertas.
\item Comunicar bloqueos de seguridad temprano.
\item Validar flujos críticos antes de la presentación.
\end{itemize}

\subsection{Seguir haciendo}
\begin{itemize}
\item Documentar reportes QA con pasos de reproducción.
\item Usar persistencia real para validar datos.
\item Mantener trazabilidad por issues.
\end{itemize}

\subsection{Menos de}
\begin{itemize}
\item Posponer configuración de entornos.
\item Redactar hallazgos sin evidencia.
\item Depender de datos residuales en pruebas.
\end{itemize}

\subsection{Dejar de hacer}
\begin{itemize}
\item Incluir credenciales hardcodeadas.
\item Tratar seguridad como actividad de cierre.
\item Cerrar HU visuales sin prueba responsive.
\end{itemize}
