\section{Introducción}
Este informe reúne los cuatro milestones funcionales del semestre y documenta la aplicación de Scrum académico, GitHub Issues, PRs, ramas, commits, métricas, trazabilidad y postmortem. Se construye con datos reales de GitHub live, Git local, anexos exportados y capturas disponibles. Si un dato no se puede verificar, se declara como pendiente.

\section{Explicación metodológica}
\subsection{Scrum académico}
El equipo trabajó con planeación por sprint, división de HU en issues, seguimiento por tablero, revisión de avances y postmortem por ciclo. Los roles fueron Scrum Master, Product Owner / Sprint Planner, Configuration Manager, DevOps Engineer y QA Lead.

\subsection{GitHub Issues y milestones}
Las HU y tareas se gestionaron con issues y se agruparon en cuatro milestones válidos. Se ignora el milestone de taller porque no representa incremento funcional del producto.

\subsection{Pull Requests, ramas y control de versiones}
El flujo combinó ramas \code{main}, \code{develop} y ramas feature. Se verifican 161 PRs, 134 mergeadas, 1 abierta y 26 cerradas sin merge. No existen releases/tags verificables.

\subsection{Trazabilidad}
La trazabilidad se documenta por sprint, milestone, issues, HU, PRs, commits y evidencias visuales. Cuando GitHub no entrega relación directa, se declara el alcance exacto del dato.

\subsection{Postmortem y estrella de mar}
El postmortem sigue la guía del profesor: evaluar producto, esfuerzo y proceso; revisar datos de calidad; evaluar roles con hechos objetivos; identificar causas; preparar acciones de mejora; y usar Starfish.

\section{Sección individual por milestone}
\subsection{Milestone 1 - Base funcional del sistema}
M1 estableció stack, autenticación, dashboard base y GitFlow. Sus métricas son 76 issues cerradas, 24 HU, 107 commits y 9 PRs. El postmortem identifica como causa raíz la ausencia de Definition of Done verificable.

\subsection{Milestone 2 - Gestión básica de flota}
M2 consolidó flota, validaciones y patrones. Sus métricas son 37 issues cerradas, 14 HU, 164 commits y 1 PR. El postmortem identifica riesgos de nombres, alias y entornos heterogéneos.

\subsection{Milestone 3 - Gestión documental y monitoreo}
M3 fortaleció documentos, monitoreo, seguridad y QA. Sus métricas son 38 issues cerradas, 14 HU, 82 commits y 0 PRs con milestone asignado. El postmortem identifica gestión tardía de variables de entorno.

\subsection{Milestone 4 - Dashboard, alertas y cierre del sistema}
M4 cerró dashboard, alertas, CI/CD, Docker, SonarCloud, SMS y evidencia final. Sus métricas son 132 issues cerradas, 20 HU, 277 commits y 3 PRs. El postmortem identifica PR abierto, falta de releases/tags y dependencia excesiva del pipeline.

\section{Comparación del semestre}
\subsection{Evolución del equipo}
El equipo pasó de una base con DoD débil a un cierre con CI/CD, SonarCloud, Docker, SMS y documentación de rúbrica. La madurez creció en QA, configuración y seguimiento.

\subsection{Tendencias}
Los sprints finales tuvieron mayor volumen de commits por cierre técnico. El número de issues cerradas aumentó en M4 por la concentración de evidencia, CI/CD, SMS y documentación final.

\subsection{Productividad}
La productividad verificable se resume en 72 HU cerradas estrictas, 283 issues cerradas en M1-M4 y 630 commits en el corte académico.

\subsection{Calidad}
La calidad evolucionó desde validación local informal hacia SonarCloud, pruebas, Docker Compose, QA con pasos reproducibles y postmortem de causas raíz.

\subsection{Participación}
La participación se evidencia con issues asignadas, commits normalizados, PRs, revisiones disponibles y roles. Ninguna métrica se usa sola como juicio individual.

\section{Postmortem general del semestre}
\subsection{Producto}
Se entregó una plataforma funcional para gestión de flota, documentos, alertas, métricas, despliegue y mensajería.

\subsection{Esfuerzo}
El esfuerzo fue alto y medible. Los roles aportaron de forma diferenciada: seguimiento Scrum, producto, configuración, DevOps y QA.

\subsection{Proceso}
El proceso mejoró de ciclo a ciclo. Los principales problemas fueron DoD incompleto, evidencia tardía, gestión de entornos y seguridad tratada tarde.

\subsection{Roles}
La evaluación se basa en hechos: issues, HU, commits, PRs, revisiones, QA, CI/CD y documentación.

\section{Retrospectiva estrella de mar general}
\subsection{Comenzar a hacer}
\begin{itemize}
\item Definir DoD con evidencia, build local y revisión funcional.
\item Automatizar exportes de GitHub al cierre de cada milestone.
\item Planear releases/tags si se requiere trazabilidad por versión.
\end{itemize}

\subsection{Más de}
\begin{itemize}
\item Revisar PRs con foco funcional y de calidad.
\item Capturar evidencia al momento de validar cada flujo.
\item Usar Scrum Poker en historias con incertidumbre.
\end{itemize}

\subsection{Seguir haciendo}
\begin{itemize}
\item Mantener issues, PRs y commits trazables.
\item Conservar postmortems por milestone.
\item Usar GitHub como fuente objetiva de seguimiento.
\end{itemize}

\subsection{Menos de}
\begin{itemize}
\item Depender de cierres masivos al final.
\item Usar métricas manuales sin criterio explícito.
\item Abrir PRs de cierre sin validación local.
\end{itemize}

\subsection{Dejar de hacer}
\begin{itemize}
\item Contar milestones de taller como evidencia funcional.
\item Presentar cifras sin separar issues, HU y PRs.
\item Postergar seguridad, QA y evidencia para el final.
\end{itemize}
