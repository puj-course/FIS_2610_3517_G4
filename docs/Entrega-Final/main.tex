\section{Introduccion}

DriveControl / AutoMinder Enterprise es un sistema academico para la gestion de flotas, documentos, alertas operativas y verificacion por mensajeria. La entrega final consolida el trabajo tecnico y metodologico desarrollado por SYNTIX TECH durante el curso Fundamentos de Ingenieria de Software.

El objetivo de este informe es centralizar la evidencia requerida por la rubrica: metricas de calidad, pruebas unitarias, coverage, despliegue reproducible, CI/CD, integracion SMS, trazabilidad Scrum y postmortem. El documento usa evidencia disponible en el repositorio y marca como pendiente cualquier captura o validacion externa que no pueda verificarse desde los archivos versionados.

El alcance de la entrega cubre el informe principal en \texttt{docs/Entrega-Final/main.tex}, las evidencias centralizadas en \texttt{docs/Entrega-Final/evidencias/} y los anexos documentales conservados en \texttt{docs/Entrega-Final/final-evaluation/}. No se modifican credenciales, archivos \texttt{.env}, codigo de aplicacion ni configuraciones funcionales del sistema.

\section{Metricas de calidad}

\section{Integracion SMS / Twilio}

\subsection{Problema inicial y causa}

El error documentado fue: \textit{Servicio SMS no configurado: faltan \texttt{TWILIO\_ACCOUNT\_SID}, \texttt{TWILIO\_AUTH\_TOKEN} o \texttt{TWILIO\_PHONE\_NUMBER}}. La causa fue que las variables necesarias se conservaban en \texttt{apps/web/.env}, mientras el backend no siempre las cargaba desde ese archivo en ejecucion local o Compose.

\subsection{Solucion}

El backend integra Twilio REST API en \texttt{backend/services/smsService.js}. El flujo permite OTP por SMS, maneja errores y registra informacion operativa enmascarada. Docker Compose carga \texttt{apps/web/.env} para el backend mediante \texttt{env\_file}, sin imprimir valores reales.

Variables requeridas, sin valores reales:

\begin{verbatim}
TWILIO_ACCOUNT_SID
TWILIO_AUTH_TOKEN
TWILIO_PHONE_NUMBER
OTP_EXPIRACION_MINUTOS
OTP_MAX_INTENTOS
OTP_COOLDOWN_SEGUNDOS
\end{verbatim}

Para pruebas academicas seguras tambien existe modo mock:

\begin{verbatim}
SMS_PROVIDER=mock
SMS_MOCK_ENABLED=true
\end{verbatim}

\subsection{Flujo Scrum/Kanban}

\begin{enumerate}[leftmargin=*]
\item Sprint planning: convertir historias en issues con criterios de aceptacion.
\item Desarrollo: ramas \texttt{feature-*} o \texttt{feature/<tema>}.
\item Integracion: commits con \texttt{Refs \#ID} o \texttt{Closes \#ID}.
\item Pull Request: resumen, validaciones, evidencia e issues relacionadas.
\item Review/CI: lint, tests, build, SonarCloud y Docker cuando aplica.
\item Cierre: merge a \texttt{develop}/\texttt{main}, actualizacion de milestone y evidencias.
\end{enumerate}

\section{Postmortem}

\subsection{Que salio bien}

\begin{itemize}[leftmargin=*]
\item La base funcional permitio cerrar el sistema con frontend, backend, MongoDB, Docker y CI/CD.
\item Las metricas propias quedaron implementadas, testeadas y documentadas.
\item Las pruebas frontend/backend validan reglas criticas y el servicio SMS sin exponer credenciales.
\item La trazabilidad final conecta issues, commits, PRs y evidencias.
\item Docker Compose facilita reproducibilidad para el evaluador mediante \texttt{make build} y \texttt{make up}.
\end{itemize}

\subsection{Lecciones aprendidas}

El cierre mostro que la calidad no se agrega al final: requiere pruebas, evidencia, configuracion reproducible, politicas de secretos y trazabilidad desde el sprint planning. Para futuros proyectos, cada historia debe cerrar con evidencia funcional, comandos reproducibles y riesgos documentados.

\section{Conclusiones}

La entrega final centraliza la evidencia principal de la rubrica y permite defender el sistema desde tres frentes: funcionamiento tecnico, calidad verificable y proceso agil. DriveControl / AutoMinder Enterprise cuenta con frontend, backend, Docker Compose, MongoDB, pruebas, SonarCloud, CI/CD y SMS/Twilio documentados.

Las principales limitaciones no estan en la existencia del codigo, sino en la completitud y manejo seguro de evidencia externa. El equipo decidio no presentar tag o release final; el control de versiones se sustenta con PRs, issues, commits y merges trazables. Las capturas SMS/Twilio deben manejarse con datos sensibles protegidos.
