\section{Introduccion}

DriveControl / AutoMinder Enterprise es un sistema academico para la gestion de flotas, documentos, alertas operativas y verificacion por mensajeria. La entrega final consolida el trabajo tecnico y metodologico desarrollado por SYNTIX TECH durante el curso Fundamentos de Ingenieria de Software.

El objetivo de este informe es centralizar la evidencia requerida por la rubrica: metricas de calidad, pruebas unitarias, coverage, despliegue reproducible, CI/CD, integracion SMS, trazabilidad Scrum y postmortem. El documento usa evidencia disponible en el repositorio y marca como pendiente cualquier captura o validacion externa que no pueda verificarse desde los archivos versionados.

El alcance de la entrega cubre el informe principal en \texttt{docs/Entrega-Final/main.tex}, las evidencias centralizadas en \texttt{docs/Entrega-Final/evidencias/} y los anexos de apoyo conservados en \texttt{docs/Entrega-Final/anexos/}.

\section{Metricas de calidad implementadas en el sistema}

Son distintas de SonarCloud porque miden calidad funcional y riesgo operativo de la flota: documentos vencidos, completitud de registros y criticidad de alertas. \section{Integracion SMS / Twilio}

\subsection{Problema inicial y causa}

El error documentado fue: \textit{Servicio SMS no configurado: faltan \texttt{TWILIO\_ACCOUNT\_SID}, \texttt{TWILIO\_AUTH\_TOKEN} o \texttt{TWILIO\_PHONE\_NUMBER}}. La causa fue que las variables necesarias se conservaban en \texttt{apps/web/.env}, mientras el backend no siempre las cargaba desde ese archivo en ejecucion local o Compose.

\subsection{Solucion}

El backend integra Twilio REST API en \path{backend/services/smsService.js}. El flujo permite OTP por SMS, maneja errores y registra informacion operativa enmascarada. Docker Compose carga \path{apps/web/.env} para el backend mediante \texttt{env\_file}, sin imprimir valores reales.

Flujo funcional defendible:

\begin{enumerate}[leftmargin=*]
\item El usuario solicita registro o recuperacion y selecciona verificacion por SMS.
\item El backend genera un codigo OTP con expiracion y limite de intentos.
\item El servicio SMS construye la solicitud a Twilio con autenticacion segura desde variables de entorno.
\item Twilio responde con estado de envio o error controlado.
\item El backend registra informacion operativa sin imprimir token, SID completo, telefono completo ni codigo OTP en logs publicos.
\end{enumerate}

Variables requeridas, sin valores reales:

\begin{verbatim}
TWILIO_ACCOUNT_SID
TWILIO_AUTH_TOKEN
TWILIO_PHONE_NUMBER
OTP_EXPIRACION_MINUTOS
OTP_MAX_INTENTOS
OTP_COOLDOWN_SEGUNDOS
\end{verbatim}

Para pruebas academicas seguras tambien existe modo mock:

\begin{verbatim}
SMS_PROVIDER=mock
SMS_MOCK_ENABLED=true
\end{verbatim}
